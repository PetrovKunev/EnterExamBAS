Big Data се дефинира чрез пет характеристики:

2. CAP теорема

2.1 Формулировка

CAP теорема (Brewer's Conjecture, 2000): Eric Brewer (UC Berkeley) представя на PODC 2000 конкретната, че разпределена система не може едновременно да гарантира трите свойства: Consistency (C), Availability (A) и Partition tolerance (P). През мрежово разделяне, системата трябва да избере между две от трите.

Gilbert & Lynch Formal Proof (2002): През 2002, Seth Gilbert и Nancy Lynch (MIT) публикуват формално математическо доказателство на CAP теоремата. Това доказателство показва, че при настъпването на мрежово разделяне (partition), система не може едновременно да поддържа консистентност (C) И наличност (A). Трябва да избере один един от двата.

Уточнение: P (Partition tolerance) е почти винаги необходимо в разпределена система, така че в практиката системите се разделят на CP (жертвуват наличност) или AP (жертвуват консистентност).

2.2 Три свойства

Consistency (Консистентност): Всички възли в системата виждат едни и същи данни в един и същи момент. При всяка операция за запис, всички четения след това виждат актуализирането.
Availability (Наличност): Системата гарантира отговор на всяка заявка (read или write) в разумно време. Няма никой узел, чиято отказ ще направи систему недостъпна.
Partition Tolerance (Толеранс към разделяне): Системата продължава да функционира дори когато мрежовото комуникиране между възлите е нарушено (например връзката между две групи възли е прекъсната).

2.3 Компромиси и класификация

При мрежово разделяне, трябва да избежаче между двата свойства:

Тип Választва Жертвува Примери Характеристики
CP Consistency, Partition tolerance Availability MongoDB, HBase, Zookeeper Може да откаже заявки при partition
AP Availability, Partition tolerance Consistency Cassandra, DynamoDB, Riak Евентуална консистентност, винаги достъпна
CA Consistency, Availability Partition tolerance RDBMS (един сървър) Работи само в един датацентър

2.4 PACELC теорема

PACELC разширение: Ако има Partition, системата избира между Availability и Consistency. Else (без partition) избира между Latency и Consistency.
  • P → A eller C: При разделяне: AP или CP системи
  • Else → L eller C: Без разделяне: търгуване между латентност и консистентност

2.5 Практичен пример

Сценарий: E-commerce платформа

Кошница (Shopping Cart): Ако избежаме консистентност (AP), потребителят вижда стари данни — не е критично. Избираме AP:高 наличност, евентуална консистентност.

Инвентар (Inventory): Важна е точност — не трябва да се продават повече елементи отколкото име. Избираме CP: консистентност първо, може да блокираме при partition.

Решение: Използваме различни системи за различни нужди: Cassandra (AP) за кошница, MongoDB (CP) за инвентар.

3. NoSQL бази данни

3.1 Key-Value хранилища

Модел: Прости двойки ключ-стойност. Суперфастни reads/writes чрез хеш таблици.
  • Redis: In-memory store, поддържа списъци, множества, хеш таблици. Идеален за cache, sessions, real-time аналитика.
  • DynamoDB: AWS managed service, автоматично scale, поддържа вторични индекси.

3.2 Document бази данни

Модел: Документи (JSON/BSON) със гъвкава схема. Поддържат вложени структури и масиви.
  • MongoDB: BSON документи, динамична схема, богата query language, поддържа транзакции.
  • CouchDB: HTTP API, master-master репликация, views за аналитика.
  • Use case: Управление на потребителски профили, CMS системи, каталози на продукти.

3.3 Column-Family хранилища

Модел: Данните се организират по колони, а не по редове. Оптимално за широки таблици и time-series данни.
  • Cassandra: Масивно скалируема, идеална за time-series (метрики, логове).
  • HBase: Построена на HDFS, поддържа billions редове и millions колони.
  • Use case: IoT сензорни данни, метрики на производителност, логове от връзки.

3.4 Graph бази данни

Модел: Възли (entities) и ребра (relationships). Оптимално за събор с висока связност.
  • Neo4j: Native graph storage, Cypher query language, ACID транзакции.
  • Use case: Social networks, препоръчвания, knowledge graphs, fraud detection.

3.8 Практичен пример: Social Network

Сценарий: Моделиране на социална мрежа

SQL подход (PostgreSQL)

Проблеми: Множество JOINs при търсене на приятели на приятелите, медленно при милиони потребители.

Предимства: Директни отношения, супербързи на близкия съсед (degrees), идеално за препоръчване.

4. MapReduce

4.1 Програмен модел

MapReduce парадигма за паралелна обработка на масивни датасети:

  1. Map фаза: Входните данни се разделят на chunks, всеки chunk обработва функцията map(key, value) която emitира междинни двойки (k', v').
  2. Shuffle & Sort: Всички двойки със един ключ k' се групират и сортират.
  3. Reduce фаза: За всяка уникална k' се вика reduce(k', [v'1, v'2, ...]) което производи финалния резултат.

4.5 Google MapReduce статия (2004)

Dean & Ghemawat, "MapReduce: Simplified Data Processing on Large Clusters"
  • Представя простата парадигма за паралелизъм без явна комуникация
  • Автоматично обработва разпределението на работа, replike на данни, отказоустойчивост
  • Демонстрира примери: grep, sort, inverted index, PageRank
  • Ключна идея: програмистът пише map() и reduce(), infrastructure се грижи за паралелизма

4.6 Apache Hadoop

HDFS (Hadoop Distributed File System):
  • NameNode: Master, управлява file system namespace, поддържа file system tree
  • DataNodes: Slaves, хранят блоковете на данни, изпращат heartbeats на NameNode
  • Block replication: По default 3 копия на всеки блок в различни машини
  • Rack awareness: 2 копия в един rack, 1 в друг rack за fault tolerance

  • Data locality: задачи се изпълняват възможно най-близо до данните

5. Разпределени файлови системи

5.1 Google File System (GFS)

Мотивация: Петабайти данни на хилядолетни дешеви сървъри, постоянни откази, това е нормално.
  • Master-slave архитектура: Един Master управлява metadata, множество ChunkServers хранят данни
  • Chunking: Файлове се разделят на 64MB chunks, всеки chunk има уникален ID
  • Replication: По default 3 копия в различни machine racks
  • Fault tolerance: Загуба на машина е очаквана, автоматична re-replication
  • Data locality: Компютите обработват данни от близки чанк сървъри

5.2 HDFS архитектура

Компонент Роля Характеристики
NameNode Управление на file system Поддържа file system tree, metadata за всеки файл. Не храни данни.
DataNode Хранилище на блокове Изпълнява block creation, deletion, replication. Сърди heartbeats.
Secondary NameNode Checkpoint периодично НЕ е backup NameNode, помага при сервизиране на metadata.

5.3 Принцип на Data Locality

Data Locality Principle: Преместване на компютъра към данните е по-евтино отколкото преместване на данните към компютъра (във възли с милионолетни записи).
  • MapReduce scheduler опитва да даде map tasks към узли където блоковете живеят
  • Ако това не е възможно, търси rack-local (на същия rack)
  • Last resort: off-rack (най-разпределено)

6. Съвременна еволюция

6.1 Apache Spark vs MapReduce

Аспект MapReduce Apache Spark
Памет Disk-based (медленно I/O) In-memory (10-100x по-бързо)
Модел изпълнение 2-stage pipeline (Map → Reduce) DAG (Directed Acyclic Graph)
Итеративни алгоритми Множество jobs, много I/O Един DAG, памет в памет, много по-бързо
API Java map/reduce функции High-level: RDD, DataFrame, SQL
Interactive queries Не добре, отнема време Отлично, REPL shell
Streaming Не (batch only) Structured Streaming, micro-batches

6.2 Lambda архитектура

Lambda архитектура (Nathan Marz, 2011): Комбинира batch layer и speed layer за истинска time-sensitive аналитика.
  • Batch layer: Hadoop/Spark обработва исторически данни на всеки N часа, produciрa batch views
  • Speed layer: Storm/Kafka обработва новите данни в реално време, produce real-time views
  • Serving layer: Merge batch views + real-time views за крайния резултат
graph TB A["Incoming data"] --> B["Batch layer"] A --> C["Speed layer"] B --> D["Batch views"] C --> E["Real-time views"] D --> F["Serving layer"] E --> F F --> G["Final result"]

6.3 Kappa архитектура

Kappa архитектура (Jay Kreps, 2014): Stream-first подход, отстранява batch layer complexity.
  • Идея: Всички данни са streams, включително историческите. Обработи с един stream processing engine (Kafka, Flink).
  • Предимства: Една логика, един engine, по-просто
  • Trade-off: Нужна е дисконтинуирана тема (replayed) за повторна обработка

6.4 Data Lake vs Data Warehouse

Аспект Data Warehouse Data Lake
Обработка Schema-on-write (обработка преди запис) Schema-on-read (обработка при четене)
Данни Чистени, структурирани, нормализирани Raw, неограничена структура, всякакво качество
Обем TB, специална hardware PB+, дешева commodity hardware
Интеграция ETL (Extract, Transform, Load) ELT (Extract, Load, Transform)
Потребители Business analysts, BI tools Data scientists, engineers, advanced analytics
Примери Oracle, Teradata, Snowflake Hadoop HDFS, S3 + Spark, Delta Lake
