1. Основни понятия

1.1 Дефиниции

База данни (БД): Организирана колекция от свързани данни, съхранявани заедно в контролирана среда, предназначена да служи на множество потребители и приложения.
СУБД (система за управление на бази данни): Софтуерна система, която позволява дефиниране, създание, поддържане и контролирано достъп до бази данни. Примери: PostgreSQL, MySQL, Oracle, SQL Server.
Система от бази данни: Съвкупност от БД + СУБД + приложения + потребители. Целта е централизиран контрол на данните с минимална техническа работа от потребителя.

1.2 Независимост на данните

Един от главните цели на СУБД е постигане на независимост между физическото и логическото представяне на данните.

Физическа независимост: Логическата структура на БД не зависи от физическото хранилище (磁 диск, SSD, брой файлове и т.н.). Промените в физическото ниво не изискват промени в приложенията.
Логическа независимост: Приложенията не зависят от точната логическа схема. Могат да бъдат добавени нови обекти без промяна на съществуващите приложения.

1.4 Схема vs инстанция

Схема (метаданни): Структурата на БД — описание на таблици, полета, типове, ограничения. Остава относително стабилна.
Инстанция (данни): Действителните стойности в БД. Изменя се при всяка операция INSERT, UPDATE, DELETE.

2. Модели на данните

2.1 Йерархичен модел

Един от най-старите модели (1960s, IMS на IBM). Представя данните като дърво.

Характеристики:
  • Един корен, родител-дете връзки
  • Едно дете може да има един родител
  • Навигация "top-down"

Пример структура:

graph TD A["Университет"] -->|има| B["Факултет"] B -->|има| C["Катедра"] C -->|има| D["Курс"] style A fill:#2563a8,color:#fff style B fill:#2563a8,color:#fff style C fill:#2563a8,color:#fff style D fill:#2563a8,color:#fff
Недостатъци: Не подържа добре много-към-много връзки; сложна навигация; слаба логическа независимост.

2.2 Мрежов модел

Разширява йерархичния модел с позволяване на графови структури (CODASYL, 1970s).

Характеристики:
  • Един записи може да има множество родители
  • Връзки като "множества" (sets) между типове записи
  • Все още навигационен подход
Недостатъци: Сложна манипулация; трудна нормализация; не е универсален стандарт.

2.3 Релационен модел

Предложен от E.F. Codd (1970). През десетилетието стана доминиращ стандарт.

Основни понятия:
  • Релация: таблица с редове и колони
  • Кортеж: ред в таблица
  • Атрибут: колона в таблица
  • Домейн: възможни стойности на атрибут
  • Степен: брой атрибути
  • Мощност: брой кортежи
Видове ключове:
  • Първичен ключ: атрибут(и), който однозначно идентифицира кортеж; NOT NULL, UNIQUE
  • Кандидат-ключ: всеки атрибут(и), който би могъл да бъде първичен
  • Външен ключ: атрибут(и), който реферира първичен ключ в друга таблица; осигурява релационна целостност
  • Съставен ключ: първичен ключ от два или повече атрибута

2.4 ER модел

Предложен от Peter Chen (1976). Концептуален модел за проектиране на БД.

Основни компоненти:
  • Обект (Entity): нещо реално или абстрактно (Student, Course)
  • Атрибут: свойство на обекта (name, age, email)
  • Връзка (Relationship): асоциация между обекти (Enrolled, Teaches)
  • Кардиналност: 1:1, 1:N (1:много), M:N (много:много)

Работен пример: Университетска база данни

ER диаграма за университетска БД

Обекти: Student, Course, Professor, Enrollment

erDiagram STUDENT ||--o{ ENROLLMENT : "enrolls in" COURSE ||--o{ ENROLLMENT : "has" PROFESSOR ||--o{ COURSE : "teaches" STUDENT { int StudentID PK string Name string Email date DateOfBirth } PROFESSOR { int ProfessorID PK string Name string Department } COURSE { int CourseID PK string Title int Credits int ProfessorID FK } ENROLLMENT { int EnrollmentID PK int StudentID FK int CourseID FK string Grade date EnrollmentDate }

Анализ на връзките:

  • Student—Enrollment: 1:N (един студент → много записвания)
  • Course—Enrollment: 1:N (един курс → много студенти)
  • Professor—Course: 1:N (един преподавател → много курсове)

2.5 Обектно-ориентиран и обектно-релационен модел

Обектно-ориентиран модел (OOP): Интегрира ООП концепции (класове, наследяване, полиморфизъм) със СУБД.
  • Съхранява обекти директно
  • Поддържа методи и наследяване
  • Примери: ObjectStore, Versant
Обектно-релационен модел (SQL:1999): Хибрид на релационния и ООП.
  • Релационна основа + допълнителни ООП функции
  • Потребителски типове данни (UDT)
  • Примери: PostgreSQL, Oracle

2.6 Документен, ключ-стойност, колонен и графов модел

Документен модел: Съхранява полуструктурирани данни (JSON, XML).
  • Пример: MongoDB, CouchDB
  • Гъвкава схема; удобно за променливи структури
Ключ-стойност модел: Проста съхраняване на двойки (ключ → стойност).
  • Пример: Redis, Memcached
  • Бързо, за cache-ване и сесии
Колонен модел: Съхранява данни по колони вместо редове.
  • Пример: HBase, Cassandra, Apache Parquet
  • Оптимално за аналитика и Big Data
Графов модел: Представя данни като възли и ребра.
  • Пример: Neo4j, ArangoDB
  • Отличен за социални мрежи, препоръчатели

Забележка: Последните четири модела се наричат "NoSQL" и се свързват с Тема 9 (Big Data).

3. Релационна алгебра

Релационна алгебра е формален система от операции над релации. Базата на SQL е интерпретация на релационна алгебра.

Основни операции

Селекция (Selection, σ): Филтрира редове според условие.
σcondition(R)
Пример: σage > 20(Students) връща всички студенти старши от 20 години.
Проекция (Projection, π): Избира определени колони.
πcol1, col2, ...(R)
Пример: πname, email(Students) връща само име и email на студентите.
Декартово произведение (Cartesian product, ×): Всеки ред от първа релация комбиниран с всеки ред от втора.
R × S
Ако |R| = m, |S| = n, тогава |R × S| = m × n редове.
Обединение (Union, ∪): Комбинира редове от две релации (еднаква схема).
R ∪ S
Дублирани редове се премахват.
Разлика (Set difference, −): Редове, които са в R но не и в S.
R − S
Естествено свързване (Natural join, ⋈): Свързва две релации по еднакви атрибути.
R ⋈ S
Еднакви колони се появяват само веднъж; се връщат само редове с съвпадащи стойности.
Theta-join (θ-join, ⋈θ): Свързване по произволно условие θ.
R ⋈θ S = σθ(R × S)
Външно свързване (Outer join): Съхранява редовете от една или две релации дори без съвпадение.
  • LEFT OUTER JOIN: Всички редове от лява релация
  • RIGHT OUTER JOIN: Всички редове от дясна релация
  • FULL OUTER JOIN: Всички редове от обе релации
Деление (Division, ÷): Намира обобщени кортежи.
R ÷ S = πX(R) \ πX((πX(R) × S) − R)
Като: "Намерете студенти, които са записани на ВСЕ курсове."

Работен пример: Релационна алгебра

Задача: Намерете имена на студенти, записани на курс 'CS101'

Дадени таблици:

Students(sid, name, age)
Enrolled(sid, cid, grade)

Решение (в релационна алгебра):

πname(σcid='CS101'(Students ⋈ Enrolled))

Поетапно:

  1. Свързване: Students ⋈ Enrolled (съхранява студенти със техните записвания)
  2. Селекция: σcid='CS101'(...) (филтрира само курс 'CS101')
  3. Проекция: πname(...) (избира само имена)

4. Нормализация

Нормализация е процес на проектиране на релационна БД, целящ да се елиминират аномалии при актуализация (INSERT, UPDATE, DELETE) и редундантност на данни.

Функционални зависимости

Функционална зависимост (FD): Атрибут Y функционално зависи от X (X → Y), ако всяка стойност на X съответства на точно една стойност на Y.

Пример: StudentID → StudentName (по един ID има един студент)

Нормални форми

1NF (First Normal Form): Всеки атрибут съдържа само атомарни (неделими) стойности. Няма повтарящи се групи.

Пример на нарушение: Колона "PhoneNumbers" съдържа (123-456, 789-012) за един студент.

2NF (Second Normal Form): 1NF + Всеки неключов атрибут пълно зависи от първичния ключ (няма частични зависимости).

Пример: Таблица (StudentID, CourseID, StudentName) нарушава 2NF, защото StudentName зависи само от StudentID, не от цялия ключ.

3NF (Third Normal Form): 2NF + Няма транзитивни зависимости между неключови атрибути.

Пример: Таблица (StudentID, DepartmentID, DepartmentName) нарушава 3NF, защото DepartmentName зависи от DepartmentID, който зависи от StudentID.

BCNF (Boyce-Codd Normal Form): 3NF + Всеки детерминант (атрибут слева от →) е кандидат-ключ.

BCNF е най-строга; обикновено 3NF е достатъчна.

Работен пример: Нормализация от UNF до 3NF

Задача: Нормализирайте таблица OrderInfo

Начално состояние (UNF):

OrderInfo(OrderID, CustomerName, CustomerCity, ProductID, ProductName, Qty, Price)

Проблеми:

  • ProductName зависи от ProductID, не от OrderID
  • Price зависи от ProductID, не от OrderID
  • CustomerCity зависи от CustomerName, което не е ключ

Стъпка 1: Преход към 1NF

Таблицата е вече в 1NF (всички стойности са атомарни).

Стъпка 2: Преход към 2NF

Елиминиране на частични зависимости. ProductID е както ProductName, Price зависят само от ProductID:

Orders(OrderID, CustomerName, CustomerCity, ProductID, Qty)
Products(ProductID, ProductName, Price)

Стъпка 3: Преход към 3NF

Елиминиране на транзитивни зависимости. CustomerCity зависи от CustomerName, който зависи от OrderID. Отделяме Customers:

Orders(OrderID, CustomerID, ProductID, Qty)
Products(ProductID, ProductName, Price)
Customers(CustomerID, CustomerName, CustomerCity)

Проверка:

  • Всяка таблица е 1NF (атомарни стойности)
  • Всеки неключов атрибут пълно зависи от първичния ключ (2NF)
  • Няма транзитивни зависимости между неключови атрибути (3NF)

Денормализация

Кога се денормализира:
  • За повишаване на производителност при четене (намаляване на JOIN операции)
  • При часто востребувани данни без актуализация
  • Кешване на калкулирани стойности
Рискове на денормализацията:
  • Редундантност на данни
  • Аномалии при актуализация
  • Инкохерентност на данни

5. SQL основи

DDL (Data Definition Language)

CREATE TABLE: Създаване на таблица.
CREATE TABLE Students ( StudentID INT PRIMARY KEY, Name VARCHAR(100) NOT NULL, Email VARCHAR(100) UNIQUE, DOB DATE );
ALTER TABLE: Изменение на структура на таблица.
ALTER TABLE Students ADD COLUMN GPA DECIMAL(3,2);
DROP TABLE: Изтриване на таблица.
DROP TABLE Students;

DML (Data Manipulation Language)

SELECT: Избор на данни.
SELECT Name, Email FROM Students WHERE Age > 20;
INSERT: Добавяне на нови редове.
INSERT INTO Students (StudentID, Name, Email) VALUES (1, 'Ivan Petrov', 'ivan@mail.bg');
UPDATE: Изменение на съществуващи редове.
UPDATE Students SET GPA = 3.8 WHERE StudentID = 1;
DELETE: Изтриване на редове.
DELETE FROM Students WHERE StudentID = 1;

JOIN операции

INNER JOIN: Редове със съвпадащи стойности в двете таблици.
SELECT s.Name, e.Grade FROM Students s INNER JOIN Enrolled e ON s.StudentID = e.StudentID;
LEFT OUTER JOIN: Всички редове от лява таблица, съответстващи от дясна.
SELECT s.Name, e.Grade FROM Students s LEFT JOIN Enrolled e ON s.StudentID = e.StudentID;
RIGHT OUTER JOIN: Всички редове от дясна таблица, съответстващи от лява.
SELECT s.Name, e.Grade FROM Students s RIGHT JOIN Enrolled e ON s.StudentID = e.StudentID;
FULL OUTER JOIN: Всички редове от обе таблици.
SELECT s.Name, e.Grade FROM Students s FULL OUTER JOIN Enrolled e ON s.StudentID = e.StudentID;
CROSS JOIN: Декартово произведение.
SELECT * FROM Students s CROSS JOIN Courses c;

GROUP BY, HAVING, подзаявки

GROUP BY и агрегатни функции:
SELECT CourseID, COUNT(*) as NumStudents, AVG(Grade) as AvgGrade FROM Enrolled GROUP BY CourseID;
HAVING: Филтър върху групирани данни.
SELECT CourseID, COUNT(*) as NumStudents FROM Enrolled GROUP BY CourseID HAVING COUNT(*) > 10;
Подзаявки (Subqueries):
SELECT Name FROM Students WHERE StudentID IN ( SELECT StudentID FROM Enrolled WHERE Grade >= 'A' );

Изгледи, индекси, тригери

Изглед (View): Виртуална таблица, дефинирана от SQL запитване.
CREATE VIEW GoodStudents AS SELECT s.Name, e.Grade FROM Students s JOIN Enrolled e ON s.StudentID = e.StudentID WHERE e.Grade >= 'B';

Предимства: Безопасност, простота, логическа независимост.

Индекс (Index): Структура за ускорение на търсене.
CREATE INDEX idx_student_email ON Students(Email);

B-tree индекси са най-често използвани.

Тригер (Trigger): Автоматично се изпълнява при определено събитие.
CREATE TRIGGER update_timestamp BEFORE UPDATE ON Students FOR EACH ROW SET NEW.UpdatedAt = CURRENT_TIMESTAMP;

Работен пример: SQL заявка

Задача: Намерете имена на студенти, записани в повече от 3 курса, подредени по брой курсове

SELECT s.Name, COUNT(e.CourseID) as NumCourses FROM Students s JOIN Enrolled e ON s.StudentID = e.StudentID GROUP BY s.StudentID, s.Name HAVING COUNT(e.CourseID) > 3 ORDER BY NumCourses DESC;

Поетапно:

  1. JOIN: Students и Enrolled по StudentID
  2. GROUP BY: Групиране по студент
  3. COUNT: Брой курсове на студент
  4. HAVING: Филтър — само студенти с повече от 3 курса
  5. ORDER BY: Сортиране по брой курсове (низходящо)

6. Транзакции и ACID

ACID свойства

Atomicity (Атомност): Транзакция е неделима — либо цялата се изпълни, либо нищо. При неуспех се откатва напълно (rollback).
Consistency (Консистентност): БД остава в валидно состояние преди и след транзакция. Всички ограничения (constraints) са удовлетворени.
Isolation (Изолация): Едновременни транзакции не се влияят взаимно. Резултатите са еквивалентни на последователно изпълнение.
Durability (Трайност): След успешно commit, данните остават съхранени дори при системен отказ.

Concurrency control

Заключване (Locking): Механизъм за предотвратяване на конфликти при едновременен достъп.
  • Shared lock (S-lock): Позволява читане; много транзакции могат да имат S-lock
  • Exclusive lock (X-lock): Позволява писане; само една транзакция може да има X-lock
Two-Phase Locking (2PL): Протокол за гарантиране на serialization.
  • Growing phase: Транзакция набира заключвания
  • Shrinking phase: Транзакция отпуска заключвания

Гарантира, че резултатът е еквивалентен на някой последователен редослед.

Recovery

Write-Ahead Logging (WAL): Преди да се напишат данни на диск, промените се записват в логов файл.
  • При отказ, системата чита логът и възстановява последното консистентно състояние
  • Гарантира durability

Parsing: Проверява синтаксиса на SQL заявката; създава синтаксисно дърво.
Validation: Проверява съответствието с метаданни (таблици, колони, типове).
Optimization: Query optimizer избира най-ефективния план за изпълнение. Използва статистика за мощност на таблици, индекси и т.н.
Compilation: Генериране на изпълним код.
Execution: Изпълнение на плана; достъп до дискови данни, кеш, изчисление на резултата.

Storage engine

Страници (Pages): Данните се организират в блокове (обикновено 4KB или 8KB). Storage engine чете/пише целиページ.
B-tree индекси: Дървовидна структура за бързо търсене. Log(N) операции за намиране, вмъкване, изтриване.
Hash индекси: Бързо точно съвпадение; не подържа range queries.

Buffer management

Buffer Pool (Cache): Кеш на дискови страници в RAM. LRU (Least Recently Used) или други политики за замяна на страници.
  • Намалява операции с диск
  • Повишава производителност

Флешкарти

Какво е база данни?
Организирана колекция от свързани данни, съхранявани заедно в контролирана среда, предназначена да служи на множество потребители и приложения.
Дефинирайте физическа независимост на данните.
Логическата структура на БД не зависи от физическото хранилище (диск, SSD и т.н.). Промените на физическото ниво не изискват промени в приложенията.
Какви са трите нива на архитектура ANSI/SPARC?
1) Външно ниво (потребителски преглед), 2) Концептуално ниво (логическа схема), 3) Вътрешно ниво (физическо представяне).
Какво е разлика между йерархичен и мрежов модел?
Йерархичен: дърво със един родител на дете; мрежов: граф с множество родители на дете.
Назовете видовете ключове в релационния модел.
Първичен (PRIMARY KEY), кандидат (един от възможните първични), външен (FOREIGN KEY), съставен (от два или повече атрибута).
Какво е функционална зависимост?
X → Y означава, че всяка стойност на X съответства на точно една стойност на Y. Пример: StudentID → StudentName.
Обяснете разликата между 2NF и 3NF.
2NF: няма частични зависимости (неключови атрибути зависят пълно от първичния ключ). 3NF: допълнително няма транзитивни зависимости между неключови атрибути.
Какво е естествено свързване (Natural join)?
Свързва две релации по еднакви атрибути. Еднакви колони се появяват само веднъж; връща редове със съвпадащи стойности.
Определете ACID свойствата на транзакции.
Atomicity: неделима; Consistency: валидна схема; Isolation: независимост от други транзакции; Durability: трайно съхранени данни.
Какво е Two-Phase Locking (2PL)?
Протокол с две фази: растяща (транзакция набира заключвания) и свиваща (отпуска заключвания). Гарантира serialization.
Какво е Write-Ahead Logging (WAL)?
Преди писане на данни на диск, промените се записват в логов файл. При отказ системата чита логът и възстановява консистентното състояние.
Обяснете разликата между INNER JOIN и LEFT OUTER JOIN.
INNER JOIN: редове със съвпадащи стойности в двете таблици. LEFT OUTER JOIN: всички редове от лява таблица + съответстващи от дясна (NULL ако няма съвпадение).
Какво е ER модел? Кои са основните компоненти?
Entity-Relationship модел (Peter Chen, 1976) за концептуално проектиране. Компоненти: обекти (entities), атрибути, връзки (relationships), кардиналност (1:1, 1:N, M:N).
Кои са нивата на изолация на транзакции?
Read Uncommitted, Read Committed, Repeatable Read, Serializable (по нарастваща строгост и намаляваща производителност).

Бърза справка

Основни дефиниции

  • БД = организирана колекция данни
  • СУБД = софтуер за управление на БД
  • Схема = структура; инстанция = данни
  • Релация = таблица; кортеж = ред; атрибут = колона

Релационна алгебра

  • σ (селекция) = филтриране редове
  • π (проекция) = избор колони
  • ⋈ (естествено свързване) = комбиниране по еднакви атрибути
  • ∪ (обединение) = комбиниране редове
  • − (разлика) = редове само в първа релация

SQL команди

  • DDL: CREATE, ALTER, DROP
  • DML: SELECT, INSERT, UPDATE, DELETE
  • JOIN: INNER, LEFT, RIGHT, FULL, CROSS
  • Групиране: GROUP BY, HAVING, ORDER BY

Нормални форми

  • 1NF: атомарни стойности
  • 2NF: 1NF + няма частични зависимости
  • 3NF: 2NF + няма транзитивни зависимости
  • BCNF: всеки детерминант е кандидат-ключ

ACID свойства

  • Atomicity: неделима операция
  • Consistency: валидна схема
  • Isolation: независимост
  • Durability: трайност
